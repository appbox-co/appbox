# ホスト型アプリサーバーと VPS: どちらを選ぶべきか

リモートサーバーを探すとき、多くの人は同じ大まかな目的に対していくつかの異なる言い方を使います。ホスト型アプリサーバー、クラウドアプリホスティング、プライベートアプリホスティング、リモートアプリホスティング、管理型アプリホスティング、VPS ホスティングなどです。

これらの言葉は重なりますが、同じものではありません。ホスト型アプリサーバーも VPS も、あなたのソフトウェアをオンラインにできます。違いは、システムのどれだけを自分で管理したいかです。

このガイドでは、特に [Plex](/apps/Plex)、[Nextcloud](/apps/Nextcloud)、[GitLab](/apps/GitLab)、[WordPress](/apps/WordPress)、File Browser、その他のセルフホストツールのようなアプリを動かしたい場合に、そのトレードオフを分かりやすく整理します。

## 短く言うと

特定のアプリをすばやくインストールして動かし、ダッシュボード操作、永続ストレージ、URL、認証情報、更新をプラットフォームのフローで扱いたいなら、**ホスト型アプリサーバー**を選びます。

仮想マシンへの完全な root アクセスが必要で、OS、パッケージ、ファイアウォール、サービス、ストレージマウント、バックアップ、トラブルシューティングを自分で管理することに慣れているなら、**VPS** を選びます。

どちらかが常に優れているわけではありません。それぞれ、異なる種類の制御に最適化されています。

## ホスト型アプリサーバーとは

ホスト型アプリサーバーは、空の OS ではなくアプリケーションを中心に設計されたサーバー環境です。

Appbox では、[アプリライブラリ](/apps)から始めるということです。アプリを選び、インストールし、ダッシュボードから管理します。Appbox は周辺のプラットフォーム要素を扱います。アプリ URL、認証情報、アプリデータ、ストレージ統合、再起動操作、バージョン、対応している場合のリソース割り当てです。

## VPS とは

VPS、つまり virtual private server は、完全な仮想マシンを提供します。通常は Linux ディストリビューションを選び、SSH で接続し、ほかのサーバーと同じようにそのマシンを管理します。

その追加の制御は本物です。カスタムパッケージをインストールし、Docker Compose を動かし、systemd サービスを設定し、ファイアウォールを調整し、ユーザーを管理し、サーバーを望む形に作れます。

Appbox も、そのレベルの制御が必要なケースのためにフル KVM VPS をサポートしています。リリースとユースケースについては、[Launch Week Day 3 - Virtual Private Servers](/blog/ja/launch-week-day-3-vps) で取り上げました。

## ホスト型アプリサーバーが向いている場面

要件が次のような場合、ホスト型アプリサーバーのほうが合うことが多いです。

- 既知のアプリをすばやくインストールしたい。
- アプリ URL、認証情報、ストレージ、再起動を1つのダッシュボードで扱いたい。
- リバースプロキシを手作業で保守することを避けたい。
- 同じストレージ環境を共有する複数のアプリを動かしたい。
- SSH から始める運用ではなく、アプリレベルの操作が欲しい。
- 下にあるサーバーをカスタマイズすることより、アプリを使うことを重視している。

たとえば [Plex](/apps/Plex)、[Nextcloud](/apps/Nextcloud)、[GitLab](/apps/GitLab)、[WordPress](/apps/WordPress) をセットアップするなら、ホスト型アプリサーバーはワークフローをアプリ中心に保ちます。同じ場所からそれぞれをインストールし、同じコントロールパネルで管理できます。

これは時間が経つほど重要になります。最初のインストールはホスティングの一部でしかありません。日々の体験は、再起動、更新、リソース確認、認証情報、ドメイン、ファイルアクセス、サポート、そしてすべてがどうつながっているかを覚えておくことです。

## VPS が向いている場面

要件が次のような場合、VPS のほうが合うことが多いです。

- root アクセスが必要。
- カスタムのシステムパッケージやカーネルレベルの挙動が必要。
- 自分の Docker Compose スタックを動かしたい。
- カスタムのネットワーク構成やファイアウォールが必要。
- systemd サービスを自分で管理したい。
- ワンクリックアプリモデルに収まらないものを構築している。

だからこそ Appbox には両方の道があります。ワンクリックのホスト型アプリは、アプリケーションのワークフローが分かっているときに最適です。VPS は、環境そのものを自由に開いておく必要があるときに最適です。

カスタムアプリケーションスタックを構築している、珍しい依存関係を動かしている、インフラを実験している場合、VPS は適切な道具になり得ます。

## 見えにくいコストは運用

最大の違いは月額料金の行ではありません。運用です。

VPS では、チェーン全体を自分で持つことになります。

- OS の更新
- パッケージリポジトリ
- ファイアウォールルール
- Docker またはサービス設定
- リバースプロキシ設定
- SSL 証明書の更新
- アプリのバックアップ
- プロセス監視
- ディスクの整理
- ログの確認
- セキュリティ強化

それを望む人もいます。必要とするチームもあります。しかし、多くのアプリホスティングのプロジェクトではそうではありません。

ホスト型アプリサーバーでは、プラットフォームが繰り返し発生するセットアップ作業の多くを取り除きます。どのアプリを選ぶか、ストレージをどう理解するか、リソースをどう見守るかは引き続き必要ですが、毎回空のシェルから始めるわけではありません。

## 実用的な判断チェックリスト

選ぶ前に、次の質問をしてみてください。

**root アクセスが必要ですか？**
必要なら VPS を選びます。不要なら、ホスト型アプリサーバーのほうがたぶん簡単です。

**アプリライブラリにある既知のアプリを動かしますか？**
はいなら、ホスト型アプリホスティングから始めてください。構成がアプリモデルを超えて大きくなったら、後から VPS を追加できます。

**SSL、プロキシルール、サービスファイルを自分で管理したいですか？**
はいなら VPS。いいえならホスト型アプリです。

**複数のアプリが同じファイルを使いますか？**
Appbox のホスト型アプリはここで役立ちます。アプリデータと共有ストレージがプラットフォームモデルの一部だからです。

**カスタムインフラを実験していますか？**
VPS を使いましょう。終わりのないシステム作業には、その形が合っています。

**Plex、Nextcloud、GitLab、WordPress をすばやく動かしたいですか？**
まずアプリライブラリを使いましょう。

## Appbox が両方を組み合わせる方法

Appbox の便利なところは、これを永続的なアイデンティティの選択として扱わなくてよいことです。アプリモデルに合うものはホスト型アプリで動かし、完全なマシン制御が必要な場面では VPS を使えます。

特に便利なのは、Appbox VPS がホスト型アプリのファイルへアクセスできることです。VPS は Appbox データを `/APPBOX_DATA/` 以下にマウントします。中には次のものがあります。

- **`/APPBOX_DATA/apps/`** - インストール済みホスト型アプリのデータ
- **`/APPBOX_DATA/storage/`** - 共有 Appbox ストレージ

つまり、メディアアプリ、ファイル同期、ダッシュボード、パスワード保管庫、Web ツールは管理型アプリフローの中に置き、カスタムサービスや開発環境は同じファイルを見られる VPS 上に置けます。

たとえば Plex や Nextcloud をホスト型アプリとして動かし、Ubuntu VPS をカスタムスクリプト、バッチ処理、バックアップ、開発ツール、その他フル Linux 制御が役立つものに使えます。VPS は別の島である必要はありません。ホスト型アプリが使うのと同じ Appbox データを扱えます。

多くの人にとって、そのハイブリッド構成は、すべてを1つのモデルへ押し込むより落ち着いて運用できます。

## ワークフローから始める

選ぶ一番簡単な方法は、サーバーの種類ではなくワークフローから始めることです。

ワークフローが「このアプリを動かしたい」なら、[Appbox アプリライブラリ](/apps)から始めます。

ワークフローが「このマシンを制御したい」なら、VPS を使います。

どちらも有効です。正しい選択は、あなたが本当に動かしたかったものにより多くの時間を使えるほうです。
